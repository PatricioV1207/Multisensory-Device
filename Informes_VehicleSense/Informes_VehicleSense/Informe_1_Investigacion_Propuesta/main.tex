\chapter*{Resumen ejecutivo}
\addcontentsline{toc}{chapter}{Resumen ejecutivo}
Este informe plantea, desde la fase inicial del proyecto, cómo diseñar un dispositivo
embarcado capaz de observar el ambiente, el movimiento y la ubicación de buses
eléctricos universitarios. La propuesta parte de un ESP32 y combina sensores de
temperatura y humedad, iluminación, presión, aceleración, velocidad angular, campo
magnético, posición y nivel acústico relativo. Los datos se validan en el borde,
se respaldan localmente y se transmiten mediante MQTT hacia una plataforma web.

La finalidad no es reemplazar sistemas certificados del vehículo, sino construir un
prototipo académico extensible para estudiar condiciones de operación, calidad del
entorno, disponibilidad de la flota y comportamiento del trayecto. Se propone una
arquitectura modular en PlatformIO, pruebas separadas por componente y un contrato de
telemetría versionado. La solución prioriza continuidad: un sensor o una conexión que
falle no debe detener el resto del dispositivo ni producir valores ficticios.

\begin{vsidea}[title={Resultado de la investigación}]
La alternativa recomendada para la primera campaña es un perfil WiFi sobre ESP32,
MQTT con TLS, microSD y página local de diagnóstico. El SIM800L se conserva como
línea experimental, pero no como transporte principal, debido a cobertura 2G,
alimentación por picos y compatibilidad TLS. La arquitectura permite migrar después a
un módem LTE sin reescribir sensores ni telemetría.
\end{vsidea}

\chapter{Planteamiento del proyecto}

\section{Contexto}
Una operación universitaria con buses eléctricos necesita información que normalmente
se encuentra dispersa: posición del vehículo, condiciones ambientales interiores,
movimiento, eventos bruscos, conectividad y estado del propio dispositivo. La
observación manual sirve para ensayos puntuales, pero no para comparar rutas, horarios
o tendencias. Un nodo IoT embarcado permite transformar esas variables físicas en una
serie temporal consultable.

El proyecto VehicleSense se formula como un prototipo de instrumentación, no como un
sistema de seguridad funcional. Sus resultados pueden apoyar investigación,
mantenimiento exploratorio y visualización operacional; cualquier decisión crítica
debe contrastarse con instrumentos calibrados y con los sistemas homologados del bus.

\section{Pregunta de investigación}
\begin{vsidea}
¿Cómo construir, con componentes accesibles y software modular, un dispositivo que
adquiera variables heterogéneas dentro de un bus eléctrico, detecte datos inválidos,
mantenga un respaldo local y publique telemetría útil sin bloquearse ante fallos
parciales?
\end{vsidea}

\section{Objetivos}
\subsection{Objetivo general}
Diseñar e implementar una arquitectura IoT multisensorial extensible para monitorear
buses eléctricos universitarios mediante adquisición local, validación, almacenamiento
y comunicación MQTT.

\subsection{Objetivos específicos}
\begin{itemize}
  \item Seleccionar sensores y buses de comunicación compatibles con el ESP32.
  \item Definir magnitudes, unidades, intervalos y criterios de validez.
  \item Separar control, sensores, almacenamiento, red y presentación en módulos.
  \item Diseñar un payload JSON versionado que omita números inválidos.
  \item Proponer diagnóstico local, actualización OTA y plataforma cloud.
  \item Elaborar una metodología reproducible de calibración e integración.
\end{itemize}

\section{Alcance inicial y exclusiones}
El prototipo medirá variables ambientales y dinámicas con componentes de bajo costo.
No accederá al CAN del vehículo, no estimará el estado de carga de la batería y no
emitirá alarmas de seguridad certificadas. Tampoco se considera en la primera fase una
calibración metrológica trazable. El audio se procesará en características agregadas;
no se almacenará voz cruda.

\section{Por qué utilizar MQTT}
MQTT es un protocolo de publicación/suscripción liviano y apropiado para entornos
restringidos \cite{mqtt}. El dispositivo no necesita conocer a todos los consumidores:
publica en temas definidos y un broker distribuye los mensajes. Esta separación
permite que el dashboard, el almacenamiento histórico y futuras reglas de alerta
evolucionen sin cambiar el firmware de adquisición.

Para la telemetría se propone QoS~1 cuando el cliente utilizado lo permita, con
identificador de mensaje y confirmación PUBACK. La microSD complementa, pero no
sustituye, esta garantía. TLS y credenciales por dispositivo son requisitos del
despliegue; MQTT en texto plano se limita al laboratorio.

\section{Qué se espera obtener}
\begin{itemize}
  \item Series de temperatura, humedad, iluminación y presión dentro del vehículo.
  \item Trayectorias, velocidad y calidad de fix del GPS.
  \item Magnitudes de aceleración, giro y campo magnético para explorar eventos.
  \item Evidencia de continuidad, reinicios, pérdida de red y recuperación.
  \item Datos exportables para informes, gráficas y análisis posteriores.
\end{itemize}

La IMU no identifica por sí sola una maniobra peligrosa. Un evento como frenado brusco
requiere definir el eje longitudinal, filtrar vibración, estimar el marco del vehículo
y validar umbrales con recorridos controlados. Del mismo modo, el nivel del micrófono
en dBFS es relativo y no equivale a dB SPL sin calibración acústica.

\chapter{Selección de hardware}

\section{Controlador ESP32}
El ESP32 integra WiFi de 2,4~GHz, múltiples UART, SPI, I2C e I2S, además de recursos
de seguridad y memoria adecuados para un nodo IoT \cite{esp32}. La placa DevKit V1
facilita programación USB y depuración. Sus limitaciones más relevantes son memoria
finita, pines de arranque y la necesidad de evitar operaciones largas dentro del loop.

\section{Sensores propuestos}
\begin{longtable}{p{3.2cm}p{3.3cm}p{3.4cm}p{4.2cm}}
\toprule
\textbf{Módulo} & \textbf{Magnitud} & \textbf{Interfaz} & \textbf{Justificación y cautela}\\
\midrule
\endhead

NEO-6M & Posición, velocidad, altitud, satélites & UART2, GPIO32/33 & NMEA ampliamente soportado; necesita cielo visible y puede tardar en fijar posición \cite{neo6}.\\
ADXL345 & Aceleración triaxial & I2C, dirección típica 0x53 & Permite medir gravedad y movimiento; requiere calibración de offset y escala \cite{adxl}.\\
L3G4200D & Velocidad angular triaxial & I2C, 0x69/0x68 & Complementa al acelerómetro; deriva y sesgo impiden obtener orientación absoluta sin fusión.\\
HMC5883L & Campo magnético & I2C, 0x1E & Útil para rumbo experimental; sensible a metal, corriente y clones QMC5883L.\\
BMP180 & Presión y temperatura interna & I2C, 0x77 & Permite estimar altitud; la presión de referencia cambia con el clima.\\
BH1750 & Iluminancia & I2C, 0x23/0x5C & Entrega lux directamente; la geometría, sombra y saturación afectan la interpretación.\\
INMP441 & Señal acústica digital & I2S & Evita entrada analógica; se propone extraer características y descartar audio crudo.\\
microSD & Respaldo local & SPI, GPIO18/19/23/5 & Conserva muestras durante cortes; requiere FAT y manejo de extracción/energía.\\
\bottomrule
\end{longtable}

El ADXL345 admite rangos seleccionables y salida digital por I2C/SPI
\cite{adxl}. El NEO-6 ofrece UART y otros buses, pero el breakout debe verificarse:
la tensión de entrada de la placa no siempre coincide con la del chip \cite{neo6}.

\section{Comunicación primaria y alternativa}
El perfil recomendado utiliza WiFi porque simplifica TLS, OTA y pruebas en campus.
El SIM800L se evalúa como experimento de conectividad GPRS, no como requisito de la
primera versión. Sus riesgos son cobertura 2G, picos de corriente cercanos a 2~A,
latencia de comandos AT y soporte TLS dependiente del firmware. La interfaz de red se
diseña para aceptar después un módem LTE como SIM7600.

Cada sensor expone un contrato equivalente a \texttt{begin()},
\texttt{update(nowMs)}, \texttt{isValid()} y \texttt{getData()}. No conoce JSON,
MQTT ni la interfaz web. La telemetría trabaja con datos tipados y el transporte
recibe un documento ya validado. Así, cambiar de WiFi a LTE no afecta el driver del
acelerómetro.

\section{Planificación temporal}
El GPS se procesa en cada iteración para no perder caracteres. IMU y audio pueden
requerir intervalos cortos; DHT11 no debe leerse más rápido que 2~s. La telemetría se
publica a un intervalo superior, por ejemplo 10~s, con una instantánea coherente. Las
reconexiones usan backoff y \texttt{millis()}, sin \texttt{delay()} prolongado.

\section{Contrato de telemetría}
El payload incluirá versión de esquema, identificadores, tiempo, valores y banderas de
validez. Los datos NaN, infinitos, obsoletos o asociados con una bandera falsa se
omiten. Un ejemplo reducido es:

\begin{lstlisting}[language={C++}]
{
  "schema_version": 3,
  "device_id": "esp32-devkit-01",
  "vehicle_id": "bus-prototipo-01",
  "uptime_ms": 123456,
  "temperature_c": 27.4,
  "dht_valid": true,
  "gps_valid": false,
  "accel_magnitude_mps2": 9.79,
  "sd_valid": true,
  "mqtt_connected": true
}
\end{lstlisting}

No se debe inventar tiempo Unix: hasta sincronizar NTP o GPS se informa uptime y
fuente temporal. Tampoco se asigna cero a una lectura ausente, porque cero podría ser
un valor físicamente válido.

\chapter{Plataforma de visualización propuesta}

\section{Página local}
El ESP32 ofrecerá un Access Point y una página sin dependencias externas. Su propósito
es diagnóstico a corta distancia: lecturas básicas, estado de microSD, WiFi, MQTT,
uptime y actualización OTA. La sección administrativa utilizará autenticación
adicional. La primera carga seguirá realizándose por USB.

La web local no reemplaza la plataforma central. Mantiene poca historia y su acceso
depende de proximidad al vehículo. Como criterio de privacidad, la propuesta inicial
evita mostrar un mapa local; la posición se reserva al sistema cloud autenticado.

\section{Plataforma cloud}
Se propone un broker MQTT, un consumidor backend persistente, PostgreSQL y una SPA.
El frontend consultará API REST para historia y WebSocket para actualización en vivo.
Las pantallas previstas son dashboard, mapa, vehículos, alertas, viajes, analítica,
reportes, dispositivos y configuración.

\chapter{Metodología de desarrollo y pruebas}

\section{Entornos PlatformIO}
La estrategia es compilar firmware especializado que no dependa de todo el hardware.
Se prevén entornos para DHT11, GPS, scanner I2C, cada chip del GY-801, BH1750,
microSD, WiFi, MQTT, audio, web local, OTA e integración completa. Las pruebas de
serialización y validación se ejecutan también como Unity nativo.

\section{Fases}
\begin{enumerate}
  \item \textbf{Inspección eléctrica:} tensiones, tierras, fuente y continuidad.
  \item \textbf{Detección:} Serial, UART, scanner I2C y montaje SPI.
  \item \textbf{Lectura individual:} rango, finitud, respuesta al estímulo y fallo.
  \item \textbf{Calibración:} acelerómetro, barómetro y criterios para magnetómetro.
  \item \textbf{Integración local:} payload, microSD, AP, API y OTA.
  \item \textbf{Comunicación:} TLS, ACL, QoS, reconexión y cola offline.
  \item \textbf{Ensayo vehicular:} recorrido controlado, repetibilidad y comparación.
\end{enumerate}

\section{Criterios de aceptación iniciales}
\begin{itemize}
  \item Todos los entornos compilan y el \texttt{main.cpp} conserva delegación mínima.
  \item La desconexión de un sensor no reinicia el equipo.
  \item El JSON nunca contiene NaN ni coordenadas ficticias.
  \item El respaldo continúa cuando MQTT no está disponible.
  \item La red y el broker se recuperan automáticamente.
  \item Credenciales reales quedan fuera del repositorio.
  \item Las pantallas distinguen datos reales, simulados y obsoletos.
\end{itemize}

\bottomrule
\end{longtable}

\chapter{Conclusiones de la propuesta}
VehicleSense es viable como plataforma académica si se construye por capas y se
documenta la calidad de cada dato. La investigación conduce a una implementación
inicial basada en ESP32, sensores digitales, microSD y WiFi/MQTT seguro. Esta opción
reduce incertidumbre antes de introducir una red celular.

El valor principal del proyecto no reside en acumular sensores, sino en asegurar que
cada medición tenga unidad, validez, contexto temporal y procedimiento de verificación.
La siguiente fase debe registrar calibraciones y pruebas físicas sin confundir salidas
de ejemplo con evidencia experimental. Después se integrarán almacenamiento,
comunicación y visualización de extremo a extremo.
